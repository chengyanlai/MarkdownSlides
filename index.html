What is Git and a little history ...

  • Free version control software created by Linus Torvalds.
  • Designed to be collaborative, and everything can be local (distributed).
  • To maintain linux kernel repository (repo) which has commits closed to 1M.
  • Sending patches $\Rightarrow$ BitKeeper $\Rightarrow$ Development of Git.
  • Installation

Things to be covered ...

Starting a local repo ......

  1. Choose a folder in which git will track everything inside.
  2. Initialize it - git init . which creates a folder .git.
  3. Stage the file(s) - git add [file].
  4. Commit to the repo - git commit.
  5. Check status and log.

What does git keep tracking? ..... good and bad practice

  • Plain text is best! codes, $\LaTeX$, markdown, html ...
  • Frequently changed binary is not good! pdf, docx, tar ...
  • Everything once committed will remain in the repo, even it is deleted afterwards.

Other tricks not mentioned ...

  • Check difference - git diff
  • Amend commit - git commit --amend
  • Configuration - git config (--global)
  • Ignore files - local .gitignore file

Branch

  • Branch is cheap in git - only changes are recorded.
  • Stay away from stable code for production.
  • Module development (issue) specific branch.
  • Up-to-date master branch.

Working with remote server ......

  • Online service - GitHub, BitBucket and GitLab.
  • Create a repo online and setup the remote at local repo
    git remote add origin (url)
    git push -u origin master
  • Get the repo somewhere else (e.g. cluster) git clone (url) [name]
  • origin is updated, but local not
    git pull
Demo ...

Things to keep in mind ...

  • Commits is local only, and required to push to remote.
  • Each branch is required to setup upstream to remote and push.
  • Sometimes pull will results in conflict.

Working with upstream (someone else) repo ......

Fork

  • You don't own the repo and you only have read access.
  • You want to contribute or simply want to implement it for your own usage.
  • Fork is one time step in the web interface.
  • Setup upstream and it allows you to update from it.
Demo ...

Make your life easier ...

  • Setup upstream remote so you can make it up-to-date with original repo.
    git remote add upstream (their url)
  • Working on a branch instead of master, so you can always pull from upstream.
    git pull upstream master
  • Regularly merge from upstream and your master branch to your working branch.
    git merge master
